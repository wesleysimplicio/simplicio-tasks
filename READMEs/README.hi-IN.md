## ⚡ TL;DR

**simplicio-tasks** एक एकल, रनटाइम-निरपेक्ष **skill** है जो किसी भी सक्षम LLM
(Claude, Codex, Copilot, Gemini, Grok, स्थानीय मॉडल) को एक **स्वायत्त लूपिंग
ऑर्केस्ट्रेटर** में बदल देता है। आप इसे किसी कार्य-भार की ओर इशारा करते हैं — *"सभी खुले
issues पूरे करो"*, *"CI कतार साफ़ करो"*, *"Jira बोर्ड खाली करो"* — और यह पूरे जीवनचक्र को
स्वयं चलाता है:

> **खोजो → समझो → निर्णय लो → कार्य करो → सत्यापित करो → सुधारो → रिकॉर्ड करो → दोहराओ**

यह किसी भी स्रोत से कार्य खोजता है, डुप्लिकेट हटाता है, आपकी मशीन के अनुसार एक एजेंट
फ़्लीट को ऑटो-स्केल करता है, प्रत्येक आइटम को एक गुणवत्ता लूप के माध्यम से लागू करता है जो
**कोड को चलाता है (केवल कंपाइल नहीं करता)**, PRs खोलता है, CI/समीक्षा फ़ीडबैक हल करता है,
मर्ज करता है, और नए कार्य के लिए **24/7** निगरानी जारी रखता है — यह सब सुरक्षा गेट्स और एक
कठोर लागत किल-स्विच के पीछे।

यह **43 नामित एक्सटेंशन पॉइंट्स** रखता है। प्रत्येक में एक हमेशा-काम-करने वाला LLM फ़ॉलबैक है,
और प्रत्येक *उपलब्ध होने पर किसी होस्ट रनटाइम के मूल कमांड से बाइंड होता है* — जिससे वह चरण
नियतात्मक और लगभग शून्य-टोकन बन जाता है। **skill किसी रनटाइम का नाम नहीं लेती; रनटाइम skill
को पहचानता है।** यही उलटाव पूरी तरकीब है: एक सार्वभौमिक प्रोटोकॉल, जिसके नीचे वैकल्पिक मूल
गति इंजेक्ट की जाती है।

```text
/simplicio-tasks termine as issues abertas
→ identity + pre-flight (kill-switch, auth, watcher)
→ discover 50 issues · dedup · build dependency DAG
→ autoscale fleet = 14 · pipeline implement→review→merge
→ each item: read body+ACs → orient code → plan → edit → run → verify → PR
→ merge · close with evidence · rollback if main breaks
→ keep polling every ~2 min for new work
```

## 🆚 caveman और rtk की तुलना में

simplicio-tasks GitHub पर दो सर्वश्रेष्ठ टोकन-बचतकर्ताओं का **गहन अध्ययन करने के बाद**
बनाया गया — [**caveman**](https://github.com/JuliusBrussee/caveman) (74k★, *बातचीत को
संपीड़ित करो*) और [**rtk**](https://github.com/rtk-ai/rtk) (63k★, *कमांड्स को संपीड़ित करो*)।
यह **दोनों** के सर्वोत्तम भागों को एक पूर्ण ऑर्केस्ट्रेटर में समेट लेता है। वे टोकन घटाते हैं;
simplicio-tasks **काम करता है** और काम करते हुए टोकन भी घटाता है।

| | 🪨 **caveman** | ⚙️ **rtk** | 🔁 **simplicio-tasks** |
|---|---|---|---|
| **यह क्या है** | Claude Code skill | Rust CLI प्रॉक्सी | रनटाइम-निरपेक्ष skill |
| **मूल विचार** | संक्षिप्त बातचीत (भराव हटाओ) | dev-कमांड आउटपुट घटाओ | **पूरे काम को ऑर्केस्ट्रेट करो** |
| **दायरा** | LLM गद्य आउटपुट | शेल कमांड आउटपुट | पूरा कार्य-जीवनचक्र, शुरू से अंत तक |
| **टोकन बचत** | जवाबों पर ~65% | कमांड्स पर 60–90% | दोनों — कैटलॉग + कैप्स + क्लैम्पिंग |
| **क्या यह काम करता है?** | ❌ केवल फ़ॉर्मेटिंग | ❌ केवल प्रॉक्सी | ✅ खोजो→लागू करो→मर्ज→बंद करो |
| **बहु-चरण स्वायत्तता** | ❌ | ❌ | ✅ निरंतर वर्कर पूल |
| **गुणवत्ता गेट्स** | — | — | ✅ AC गेट · रन-सत्यापन · प्रतिकूल सत्यापन · डिलीवरी गेट |
| **सुरक्षा** | — | semgrep, अस्वीकरण | ✅ 4-अवस्था निर्णय · प्रमाणन · सीक्रेट-स्कैन · मानव गेट · किल-स्विच |
| **24/7 लूप** | ❌ | ❌ | ✅ टिकाऊ watcher, स्व-उपचारी |
| **रनटाइम बाइंडिंग** | Claude/Codex/Gemini | कोई भी (PATH प्रॉक्सी) | **कोई भी** (43 एक्सटेंशन पॉइंट्स) |
| **हमने क्या लिया** | संक्षिप्त वर्कर रिपोर्ट्स, घनत्व स्तर, never-paraphrase गार्ड, ईमानदार बेसलाइन | प्रति-कमांड घटाव कैटलॉग, सिग्नल-स्तरीय कैप्स, कंपाउंड-क्लैम्पिंग, fail-open, 4-अवस्था निर्णय | — |
| **हमने क्या छोड़ा** | व्याकरण शब्द-लोपन (कोड गुणवत्ता घटाता है) | प्रति-भाषा रजिस्ट्रियाँ (रनटाइम-विशिष्ट) | — |

> हमने जानबूझकर caveman के "गुफावासी की तरह बोलो" शब्द-लोपन को **अस्वीकार** किया — संक्षिप्त
> *गद्य* ठीक है, पर व्याकरण को बिगाड़ना कोड और पुष्टियों को घटाता है। हमने *अनुशासन* रखा
> (कोड/URLs/पाथ्स को कभी पैराफ़्रेज़ न करना), न कि वह तिकड़म।

## 🧩 43 एक्सटेंशन पॉइंट्स

कार्य का प्रत्येक चरण एक **नामित एक्सटेंशन पॉइंट** पर होता है। यदि कोई होस्ट रनटाइम एक मूल
क्षमता प्रदान करता है तो वह **बाइंड** हो जाता है (नियतात्मक, लगभग शून्य टोकन)। अन्यथा LLM
मानक उपकरणों (शेल, git, gh, फ़ाइल एडिट, वेब) के साथ **फ़ॉलबैक** करता है। skill अमूर्तन पर
निर्भर करती है, किसी विशिष्ट रनटाइम पर कभी नहीं।

### ऑर्केस्ट्रेशन और स्केल
| पॉइंट | यह क्या करता है |
|---|---|
| `orient` | संपीड़ित रेपो/कार्य मानचित्र |
| `normalize` | कार्य-आइटम → कैनोनिकल स्कीमा |
| `intake` | किसी स्प्रिंट/बोर्ड लिंक से कार्य ग्रहण करना |
| `source_adapter` | समरूप स्रोत कनेक्टर (list/get/claim/update/attach/close) |
| `autoscale` | मशीन प्रोफ़ाइल से सुरक्षित फ़्लीट आकार |
| `plan` / `decide` | योजना और निर्णय समर्थन |
| `execute` | बड़े/यांत्रिक कार्य के लिए स्थानीय एजेंट फ़ैन-आउट |
| `issue_factory` | पूर्ण लूप: discover→claim→implement→PR |
| `claim` | परमाणविक, क्रॉस-सेशन-सुरक्षित कार्य-आइटम क्लेम |
| `worktree` | प्रति-आइटम पृथक चेकआउट |
| `dependency_graph` | आइटम्स के बीच पुनरारंभ-योग्य DAG क्रमबद्धता |
| `durable_workflow` | प्रति-आइटम पाइपलाइन एक पुनरारंभ-योग्य चरण स्टेट-मशीन के रूप में |
| `work_queue` | ऑटो-रीट्राई + राइट-लॉक के साथ टिकाऊ प्राथमिकता कतार |
| `resource_governor` | गतिशील मध्य-लूप थ्रॉटल + मशीन-स्तर सीमाएँ |
| `model_route` | प्रति उप-कार्य सबसे सस्ता व्यवहार्य सब्सट्रेट (L0→remote) |
| `model_preflight` | जनरेशन रूट करने से पहले एक प्रयोग-योग्य मॉडल जाँचना |

### एडिटिंग, गुणवत्ता और साक्ष्य
| पॉइंट | यह क्या करता है |
|---|---|
| `deterministic_edit` | किसी निर्णीत परिवर्तन का यांत्रिक, शून्य-टोकन अनुप्रयोग |
| `diagnostics` | बिल्ड/टेस्ट आउटपुट पार्स करना → संरचित त्रुटियाँ → पुनरावृत्ति |
| `toolchain_detect` | रेपो के वास्तविक build/lint/typecheck/test स्टैक का पता लगाना |
| `validate` / `smoke` | रन-सत्यापन: "काम करता है, केवल कंपाइल नहीं होता" |

| `endpoint_compare` | वेब↔API↔एजेंट विचलन → फ़ॉलो-अप आइटम्स |

| `retry` | विफलता वर्ग के अनुसार वर्गीकृत रीट्राई+बैकऑफ़ |
| `reuse_precedent` | किसी पूर्व-हल किए रन से मिलान → पुनः उपयोग, पुनर्जनन नहीं |
| `trajectory` | स्व-सुधार हेतु रन परिणाम रिकॉर्ड करना |
| `learn` | किसी रन से सीखना — पूर्व-उदाहरण/स्मृति अपडेट करना |
| `status` | लाइव अवलोकनीयता डैशबोर्ड |
| `capability_rank` | किसी उप-कार्य के लिए कौन-सी skill/उपकरण उपयुक्त है, उसकी रैंकिंग |

## 📦 अंदर सब कुछ

skill जो कुछ रखती है उसकी एक पूर्ण सूची — प्रत्येक तंत्र, उद्धृत।

### लूप (7 चरण + उप-चरण)
- **चरण 0** — अनुबंध लोड करें (कैनोनिकल प्रोटोकॉल)।
- **चरण 1** — पहचान + सस्ता पर्यावरण पता लगाना।
- **चरण 1b** — 43 एक्सटेंशन पॉइंट्स (मूल बाइंड या LLM-फ़ॉलबैक)।
- **चरण 1c** — टोकन-अर्थव्यवस्था गेट: `THINK / NO-THINK`, `INTERNET off by default`,
  `terminal-first execution`, **आउटपुट-घटाव कैटलॉग**, **सिग्नल-स्तरीय कैप्स**,
  **success-collapse + dedup**, **कंपाउंड-कमांड क्लैम्पिंग**, **उपभोक्ता-राउटेड
  घनत्व स्तर**, **fail-open**, **auto-clarity (सुरक्षा संक्षिप्तता को ओवरराइड करती है)**।
- **चरण 1d** — प्री-फ़्लाइट: किल-स्विच बजट, स्रोत प्रमाणन, watcher को सक्रिय करना।
- **चरण 2** — कार्य-आइटम खोजना + सामान्यीकृत करना (कोई भी स्रोत एडाप्टर)।
- **चरण 2b** — गहन intake: पूरा बॉडी + टिप्पणियाँ पढ़ना, **स्वीकृति
  मानदंड** निकालना, **कोडबेस को ओरिएंट करना**, **signatures-only रीड मोड**, एक योजना बनाना।
- **चरण 2c** — निर्भरता DAG + टोपोलॉजिकल शेड्यूलिंग।
- **चरण 3** — द्वि-पथ राउटर: **fast-path** बनाम **heavy-path** निरंतर वर्कर
  पूल · **conflict-aware isolation** · **worker report contract** · **corrections
  memory**।
- **चरण 3b** — निरंतर intake: इंट्रा-रन पोलर + निष्क्रिय watcher (किसी भी मिनट नया कार्य देखें)।
- **चरण 3c** — गति मॉडल: पाइपलाइन (बैरियर नहीं), साझा कंपाइल कैश,
  verify-once-at-merge, **साझा संदर्भ डाइजेस्ट**।
- **चरण 3d** — मॉडल राउटिंग L0→L4 (नियतात्मक → स्थानीय → मध्य → तर्कणा → सशुल्क)।
- **चरण 4** — गुणवत्ता लूप · **AC गेट (वास्तविक DoD)** · **रन-सत्यापन** ·
  **प्रतिकूल बहु-मत सत्यापन** · **स्थैतिक-विश्लेषण गेट**।
- **चरण 5** — सुरक्षा गेट्स: सीक्रेट-स्कैन, अपरिवर्तनीय-संचालन मानव गेट, **4-अवस्था
  पूर्व-निष्पादन निर्णय**, **प्रति-खंड कंपाउंड प्रमाणन**, **trust-before-load
  कॉन्फ़िग**, **सप्लाई-चेन अखंडता गेट**, **transform_guard**।
- **चरण 6** — डिलीवर + बंद करना + स्व-ऑडिट · **साक्ष्य पैकेज** · **वास्तविकता
  सत्यापित करें (स्व-रिपोर्ट पर कभी भरोसा न करें)** · **rollback-guard यदि मर्ज main को तोड़े**।
- **चरण 6b** — फ़ीडबैक लूप बंद करना: CI → फ़िक्स, समीक्षा टिप्पणियाँ → हल,
  branch-behind → सुलह, पूर्ण **PR जीवनचक्र** मर्ज-तैयार होने तक।
- **चरण 7** — 24/7 स्थायी लूप (10 अक्ष): टिकाऊ ड्राइवर, पूर्ण कवरेज मैट्रिक्स,
  टिकाऊ स्थिति, **लागत शासन + कठोर किल-स्विच**, अनिगरानी सुरक्षा,
  स्व-उपचार + **विफलता वर्ग द्वारा बुद्धिमान रीट्राई**, प्राथमिकता/WIP,
  अवलोकनीयता + **आवधिक बचत ऑडिट** + **स्नैपशॉट मापन**,
  स्व-सुधार, समन्वय और स्वच्छ स्टॉप।

### टोकन अर्थव्यवस्था (rtk + caveman से समाहित)
- टर्मिनल-फ़र्स्ट निष्पादन — किसी कमांड का कभी अनुकरण न करें।
- **क्रॉस-प्लेटफ़ॉर्म** प्रतिस्थापन तालिका (Windows / macOS / Linux): 30+ तथ्य जिन्हें
  टर्मिनल LLM से सस्ते में उत्तर देता है।
- **आउटपुट-घटाव कैटलॉग** डेटा के रूप में: प्रति-कमांड रेसिपी, अपेक्षित-बचत %,
  `skip-if-structured` गार्ड।
- **सिग्नल-स्तरीय कैप्स**: `CAP_ERRORS / CAP_WARNINGS / CAP_LIST / CAP_INVENTORY`।
- **Success-collapse** + **dedup-with-counts** (एक `unless errors` गार्ड के साथ)।
- **कंपाउंड-कमांड क्लैम्पिंग** — प्रति-खंड, pipe/redirect-सुरक्षित, fail-open।
- **उपभोक्ता द्वारा घनत्व स्तर** (मशीन बनाम मानव); पहले से सघन सामग्री छोड़ना।
- **वर्कर रिपोर्ट अनुबंध** — उप-एजेंट्स के लिए status-token-first संक्षिप्त स्कीमा।
- **ईमानदार बचत बेसलाइन** = यथार्थवादी नियंत्रण भुजा, **एक पास होते गुणवत्ता गेट से
  बंधी** (वह संपीड़न जो अपना गेट विफल कर दे, शून्य श्रेय पाता है)।

## 🚀 इंस्टॉल करें और उपयोग करें

simplicio-tasks एक **skill** है — एक एकल फ़ोल्डर जिसे आप किसी भी ऐसे रनटाइम में डाल देते हैं
जो skills लोड करता है। कोई निर्भरता नहीं, किसी बाइनरी की आवश्यकता नहीं।

```bash
# Claude Code (project or user skills dir)
git clone https://github.com/wesleysimplicio/simplicio-tasks
cp -r simplicio-tasks/.claude/skills/simplicio-tasks  <your-repo>/.claude/skills/

# then, in your agent:
/simplicio-tasks finish all the open issues
```

अन्य रनटाइम (Codex, Gemini, Copilot, स्थानीय एजेंट) वही
`SKILL.md` लोड करते हैं — प्रति-रनटाइम प्रवेश-बिंदुओं के लिए [`AGENTS.md`](../AGENTS.md),
[`CLAUDE.md`](../CLAUDE.md) और [`GEMINI.md`](../GEMINI.md) देखें। जहाँ कोई होस्ट रनटाइम मूल
कमांड्स प्रदान करता है, वह उन्हें एक्सटेंशन पॉइंट्स से स्वतः बाइंड कर देता है; अन्यथा LLM
फ़ॉलबैक्स **100%** कार्य को कवर करते हैं।

**किसी अनिगरानी 24/7 रन से पहले:** एक लागत सीमा सेट करें (`.orchestrator/loop-budget.json`,
`daily_usd_ceiling > 0`), पुष्टि करें कि स्रोत प्रमाणन स्थायी है, और अपरिवर्तनीय-संचालन
मानव गेट + सीक्रेट-स्कैन चालू रखें। `ceiling = 0` के साथ watcher अनिगरानी चलने से इनकार
कर देता है (fail-safe)।
