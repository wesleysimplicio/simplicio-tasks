## ⚡ TL;DR

**simplicio-tasks** — это единый, не зависящий от среды выполнения **навык (skill)**,
который превращает любую сильную LLM (Claude, Codex, Copilot, Gemini, Grok, локальные
модели) в **автономный циклический оркестратор**. Вы указываете ему на объём работы —
*«закрой все открытые issue»*, *«разгреби очередь CI»*, *«опустоши доску Jira»* — и он
самостоятельно прогоняет весь жизненный цикл:

> **обнаружить → понять → решить → действовать → проверить → исправить → зафиксировать → повторить**

Он обнаруживает работу из любого источника, устраняет дубликаты, автоматически
масштабирует флот агентов под вашу машину, реализует каждый пункт через цикл
обеспечения качества, который **запускает код (а не просто компилирует его)**, открывает
PR, разрешает замечания CI/ревью, выполняет слияние и продолжает следить за новой работой
**24/7** — всё это за предохранительными воротами и жёстким аварийным выключателем
расходов.

Он несёт в себе **43 именованные точки расширения**. У каждой есть всегда работающий
LLM-фолбэк, и каждая *привязывается к нативной команде хост-среды*, когда таковая
присутствует — делая шаг детерминированным и почти бестокенным. **Навык не называет
ни одну среду выполнения; среда выполнения обнаруживает навык.** В этой инверсии и кроется
весь фокус: единый универсальный протокол с опциональной нативной скоростью, подставленной
снизу.

```text
/simplicio-tasks termine as issues abertas
→ identity + pre-flight (kill-switch, auth, watcher)
→ discover 50 issues · dedup · build dependency DAG
→ autoscale fleet = 14 · pipeline implement→review→merge
→ each item: read body+ACs → orient code → plan → edit → run → verify → PR
→ merge · close with evidence · rollback if main breaks
→ keep polling every ~2 min for new work
```

## 🆚 в сравнении с caveman и rtk

simplicio-tasks был создан **после глубокого изучения** двух лучших инструментов
экономии токенов на GitHub — [**caveman**](https://github.com/JuliusBrussee/caveman)
(74k★, *сжать разговор*) и [**rtk**](https://github.com/rtk-ai/rtk) (63k★, *сжать
команды*). Он соединяет лучшее из **обоих** в полноценном оркестраторе. Они сокращают
токены; simplicio-tasks **делает работу** и сокращает токены в процессе её выполнения.

| | 🪨 **caveman** | ⚙️ **rtk** | 🔁 **simplicio-tasks** |
|---|---|---|---|
| **Что это** | Навык Claude Code | Rust CLI-прокси | Навык, не зависящий от среды |
| **Основная идея** | Говорить короче (убрать «воду») | Сократить вывод dev-команд | **Оркестрировать всю работу целиком** |
| **Охват** | Прозаический вывод LLM | Вывод shell-команд | Полный жизненный цикл работы, от и до |
| **Экономия токенов** | ~65% на ответах | 60–90% на командах | И то, и другое — каталог + лимиты + ограничение |
| **Делает ли работу?** | ❌ только форматирование | ❌ только прокси | ✅ обнаружить→реализовать→слить→закрыть |
| **Многошаговая автономность** | ❌ | ❌ | ✅ непрерывный пул воркеров |
| **Ворота качества** | — | — | ✅ ворота AC · проверка запуском · состязательная проверка · ворота поставки |
| **Безопасность** | — | semgrep, дисклеймеры | ✅ вердикт из 4 состояний · аттестация · скан секретов · человеческие ворота · аварийный выключатель |
| **Цикл 24/7** | ❌ | ❌ | ✅ устойчивый watcher, самовосстановление |
| **Привязка к среде** | Claude/Codex/Gemini | любая (PATH-прокси) | **любая** (43 точки расширения) |
| **Что мы взяли** | лаконичные отчёты воркеров, уровни плотности, защита «никогда не перефразировать», честный базовый уровень | каталог сокращения по каждой команде, лимиты по уровням сигнала, ограничение составных команд, fail-open, вердикт из 4 состояний | — |
| **Что мы оставили в стороне** | грамматическое выбрасывание слов (ухудшает качество кода) | реестры по языкам (привязаны к конкретной среде) | — |

> Мы **намеренно отказались** от приёма caveman «говорить как пещерный человек» с
> выбрасыванием слов — лаконичная *проза* — это нормально, но коверканье грамматики
> ухудшает код и подтверждения. Мы сохранили *дисциплину* (никогда не перефразировать
> код/URL/пути), но не трюк.

## 🧩 43 точки расширения

Каждый шаг работы происходит в **именованной точке расширения**. Если хост-среда
предоставляет нативную возможность, точка **привязывается** к ней (детерминированно,
почти без токенов). В противном случае LLM выполняет **фолбэк** стандартными
инструментами (shell, git, gh, редактирование файлов, веб). Навык зависит от
абстракции, а не от конкретной среды выполнения.

### Оркестрация и масштаб
| Точка | Что она делает |
|---|---|
| `orient` | Сжатая карта репозитория/работы |
| `normalize` | Рабочий элемент → канонической схеме |
| `intake` | Приём работы по ссылке на спринт/доску |
| `source_adapter` | Единый коннектор к источнику (list/get/claim/update/attach/close) |
| `autoscale` | Безопасный размер флота из профиля машины |
| `plan` / `decide` | Поддержка планирования и принятия решений |
| `execute` | Локальный веер агентов для массовой/механической работы |
| `issue_factory` | Полный цикл: discover→claim→implement→PR |
| `claim` | Атомарный, безопасный для разных сессий захват рабочего элемента |
| `worktree` | Изолированный чекаут для каждого элемента |
| `dependency_graph` | Возобновляемое упорядочивание DAG между элементами |
| `durable_workflow` | Конвейер для каждого элемента как возобновляемый автомат фаз |
| `work_queue` | Устойчивая приоритетная очередь с авто-повтором + блокировкой записи |
| `resource_governor` | Динамический троттлинг внутри цикла + потолки по классу машины |
| `model_route` | Самая дешёвая жизнеспособная основа для каждой подзадачи (L0→удалённая) |
| `model_preflight` | Проба пригодной модели перед маршрутизацией генерации |

### Редактирование, качество и доказательства
| Точка | Что она делает |
|---|---|
| `deterministic_edit` | Механическое, бестокенное применение принятого изменения |
| `diagnostics` | Парсинг вывода сборки/тестов → структурированные ошибки → итерация |
| `toolchain_detect` | Определение реального стека сборки/линтинга/типизации/тестов репозитория |
| `validate` / `smoke` | Проверка запуском: «работает, а не просто компилируется» |
| `delivery_gate` | DoD: проверка AC + регрессия + ревью диффа + сертификат |
| `endpoint_compare` | Расхождение Web↔API↔агент → последующие элементы |
| `web_verify` | Управление реальным браузером для подтверждения работы UI-изменения |
| `pr` / `evidence` | Открытие/обновление PR + проверяемый журнал доказательств |
| `retry` | Классифицированный повтор + backoff по классу сбоя |
| `reuse_precedent` | Сопоставление с ранее решённым прогоном → переиспользовать, а не генерировать заново |
| `trajectory` | Запись результата прогона для самосовершенствования |
| `learn` | Обучение на прогоне — обновление прецедентов/памяти |
| `status` | Живая панель наблюдаемости |
| `capability_rank` | Ранжирование, какой навык/инструмент подходит для подзадачи |

### Токены, контекст и безопасность
| Точка | Что она делает |
|---|---|
| `recall` | Прежние решения / прецеденты |
| `compress` | Сжатие контекста / ограничение вывода |
| `prompt_budget` | Конверт промпта с бюджетом токенов + кэш фрагментов |
| `shell_exec` | Ограниченное выполнение shell (структурированное, с границами) |
| `transform_guard` | Проверка, что компактизация сохранила каждый токен кода/URL/пути/версии |
| `action_gate` | Классификация риска каждой мутации (safe/auto/ask) перед запуском |
| `security` | Скан цепочки поставок / секретов |
| `human_gate` | Асинхронный канал человеческого одобрения |
| `notify` | Отправка прогресса/блокеров/дайджеста + приём одобрений |
| `checkpoint_restore` | Снимок состояния перед рискованной партией; восстановление при сбое |
| `watcher` | Устойчивый планировщик / поллер (переживает перезагрузку) |
| `savings_ledger` | Реальный учёт расхода токенов за сессию |
| `web_research` | Получение актуального внешнего знания, под воротами, с провенансом |

## 📦 Всё, что внутри

Полная опись того, что несёт в себе навык — каждый механизм, со ссылкой.

### Цикл (7 шагов + подшаги)
- **Шаг 0** — Загрузка контракта (канонический протокол).
- **Шаг 1** — Идентичность + дешёвое определение окружения.
- **Шаг 1b** — 43 точки расширения (привязка к нативу или LLM-фолбэк).
- **Шаг 1c** — Ворота экономии токенов: `THINK / NO-THINK`, `INTERNET off by default`,
  `terminal-first execution`, **каталог сокращения вывода**, **лимиты по уровням сигнала**,
  **схлопывание успеха + дедуп**, **ограничение составных команд**, **уровни плотности по
  потребителю**, **fail-open**, **авто-ясность (безопасность важнее краткости)**.
- **Шаг 1d** — Pre-flight: бюджет аварийного выключателя, аутентификация источника,
  взвод watcher'а.
- **Шаг 2** — Обнаружение + нормализация рабочих элементов (любой адаптер источника).
- **Шаг 2b** — Глубокий приём: чтение полного тела + комментариев, извлечение
  **критериев приёмки**, **ориентация в кодовой базе**, **режим чтения только сигнатур**,
  построение плана.
- **Шаг 2c** — DAG зависимостей + топологическое планирование.
- **Шаг 3** — Двухпутевой маршрутизатор: непрерывный пул воркеров **быстрого пути** против
  **тяжёлого пути** · **изоляция с учётом конфликтов** · **контракт отчёта воркера** ·
  **память исправлений**.
- **Шаг 3b** — Непрерывный приём: внутрипрогонный поллер + watcher простоя (увидеть новую
  работу в любую минуту).
- **Шаг 3c** — Модель скорости: конвейер (не барьер), общий кэш компиляции,
  проверка-один-раз-при-слиянии, **общий дайджест контекста**.
- **Шаг 3d** — Маршрутизация моделей L0→L4 (детерминированная → локальная → средняя →
  рассуждающая → платная).
- **Шаг 4** — Цикл качества · **ворота AC (настоящий DoD)** · **проверка запуском** ·
  **состязательная проверка с мультиголосованием** · **ворота статического анализа**.
- **Шаг 5** — Предохранительные ворота: скан секретов, человеческие ворота для
  необратимых операций, **вердикт из 4 состояний перед выполнением**, **аттестация
  составных команд посегментно**, **конфигурация «доверять перед загрузкой»**, **ворота
  целостности цепочки поставок**, **transform_guard**.
- **Шаг 6** — Поставка + закрытие + самоаудит · **пакет доказательств** · **проверка
  реальности (никогда не доверять самоотчёту)** · **ворота отката, если слияние ломает
  main**.
- **Шаг 6b** — Замыкание цикла обратной связи: CI → исправить, комментарии ревью →
  разрешить, отставание ветки → согласовать, полный **жизненный цикл PR** до готовности к
  слиянию.
- **Шаг 7** — Постоянный цикл 24/7 (10 осей): устойчивый драйвер, матрица полного
  покрытия, устойчивое состояние, **управление расходами + жёсткий аварийный выключатель**,
  безопасность без присмотра, самовосстановление + **интеллектуальный повтор по классу
  сбоя**, приоритизация/WIP, наблюдаемость + **периодический аудит экономии** + **измерение
  по снимку**, самосовершенствование, координация и чистая остановка.

### Экономия токенов (вобрано из rtk + caveman)
- Выполнение в первую очередь в терминале — никогда не симулировать команду.
- **Кросс-платформенная** таблица замен (Windows / macOS / Linux): 30+ фактов, на которые
  терминал отвечает дешевле, чем LLM.
- **Каталог сокращения вывода** как данные: рецепт для каждой команды, ожидаемая экономия %,
  защита `skip-if-structured`.
- **Лимиты по уровням сигнала**: `CAP_ERRORS / CAP_WARNINGS / CAP_LIST / CAP_INVENTORY`.
- **Схлопывание успеха** + **дедуп со счётчиками** (с защитой `unless errors`).
- **Ограничение составных команд** — посегментно, безопасно к пайпам/редиректам, fail-open.
- **Уровни плотности по потребителю** (машина против человека); пропуск уже плотного
  контента.
- **Контракт отчёта воркера** — лаконичная схема с токеном статуса в начале для субагентов.
- **Честный базовый уровень экономии** = реалистичная контрольная группа, **привязанная к
  пройденным воротам качества** (сжатие, провалившее свои ворота, получает ноль зачёта).

### Качество и поставка
- Чек-лист DoD по критериям приёмки · проверка запуском · состязательная проверка ·
  ворота статического анализа · сертификат поставки · повторная проверка реальности ·
  автоматический откат.

### Безопасность
- Скан секретов · человеческие ворота для необратимых операций · вердикт из 4 состояний
  (никогда не повышать привилегии) · аттестация составных команд · доверять-перед-загрузкой ·
  целостность цепочки поставок · защита от prompt-инъекций · жёсткий $-аварийный выключатель
  для прогонов без присмотра.

### Автономность 24/7
- Устойчивый планировщик · живая очередь + watcher простоя · устойчивый журнал/состояние ·
  предохранители (circuit breakers) · карантин dead-letter · самосовершенствование и
  мета-ревью · атомарные захваты на нескольких инстансах · чистый сигнал STOP.

## 🚀 Установка и использование

simplicio-tasks — это **навык**: единая папка, которую вы кладёте в любую среду
выполнения, загружающую навыки. Никаких зависимостей, никакого бинарника не требуется.

```bash
# Claude Code (project or user skills dir)
git clone https://github.com/wesleysimplicio/simplicio-tasks
cp -r simplicio-tasks/.claude/skills/simplicio-tasks  <your-repo>/.claude/skills/

# then, in your agent:
/simplicio-tasks finish all the open issues
```

Другие среды выполнения (Codex, Gemini, Copilot, локальные агенты) загружают тот же
`SKILL.md` — см. [`AGENTS.md`](../AGENTS.md), [`CLAUDE.md`](../CLAUDE.md) и
[`GEMINI.md`](../GEMINI.md) для точек входа по каждой среде. Там, где хост-среда
предоставляет нативные команды, она автоматически привязывает их к точкам расширения; в
противном случае LLM-фолбэки покрывают **100%** работы.

**Перед прогоном 24/7 без присмотра:** установите потолок расходов
(`.orchestrator/loop-budget.json`, `daily_usd_ceiling > 0`), убедитесь, что
аутентификация источника постоянна, и держите включёнными человеческие ворота для
необратимых операций + скан секретов. При `ceiling = 0` watcher отказывается работать без
присмотра (отказоустойчиво).

## 📊 Экономия токенов

Каждое сообщение завершается честной строкой экономии:

```
simplicio-tasks: ~<spent> tokens · baseline ~<control-arm> · saved ~<saved> (<pct>%)
```

Базовый уровень — это **самый дешёвый разумный неоркестрированный путь** к тому же
результату, а не многословное чучело — и экономия **засчитывается, только когда проходят
проверка запуском элемента и ворота критериев приёмки**. Сырое сжатие само по себе никогда
не считается успехом.
